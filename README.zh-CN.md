# Semantic Alignment

[English README](README.md)

## 它解决什么问题

很多项目并不会完全按照最初设想推进。

在设计、实现或写作过程中，方向经常会发生变化。有些变化来自更好的方案，有些则来自现实约束：工具暂时不可用、时间不足、素材缺失、权限未开放，或需求本身还不够清晰。

真正需要警惕的是后者。临时妥协在当下可能是合理的，但时间一长，很容易变成默认路线。等限制条件消失时，项目仍可能沿着旧的妥协继续推进，只是因为当初绕路的原因已经被遗忘。

AI agent 还会带来另一类风险：它会自动补全缺失的细节。有些补全是有帮助的，有些则可能改变产品含义、交互方向、架构选择或验收标准。如果没有明确记录，之后很难区分哪些来自用户，哪些来自 agent 推断，也很难判断最终产物是否仍然服务于原始目标。

`semantic-alignment` 用来让这些语义保持可见。

## 它怎么解决

这个 skill 会让 agent 为项目维护一组语义记录。

它重点记录：

- 用户当前的目标和设计意图
- 意图发生过哪些重要变化，以及变化原因
- 哪些路线是因为限制条件才被临时接受的
- 什么条件出现时，需要重新检查旧妥协
- agent 补充了哪些用户没有明确说明的设计
- 最终代码、设计或文档是否仍然符合这些语义

用户主要查看 `user-semantics.md`。它只呈现当前语义状态，保持清晰、可读，不把完整历史堆在一起。详细变化、检查和审计材料放在其他文件中。

## 它会提醒哪些事

例如：

- 当初因为无法导出文件，所以采用复制到剪贴板；现在文件导出可用了，是否需要回到原路线？
- 用户原本需要一个安静、实用的工具，但新请求正在把设计推向强营销风格页面。
- agent 增加了自动保存、快捷键或导航结构，但这些并不是用户明确要求的。
- 某个局部实现本身可行，但已经偏离整体目标。
- 真实代码或设计，与 agent 认为自己实现的语义并不一致。

它不会记录每一句对话，只记录会影响项目和设计方向的语义。

## 工作方式

在重要任务开始前，agent 会先读取当前语义框架，再进行规划或修改。

这个 skill 将对齐检查分成三层：

1. **用户语义**：用户当前想要什么。
2. **实现语义**：agent 理解用户后，打算在产物中实现什么。
3. **产物检查**：真实代码、设计、文档或输出是否符合这些实现语义。

当语义发生重要变化时，agent 会记录这是新增、更新还是删除，并记录原因。如果变化由限制条件导致，还可以留下重新检查的触发条件。等条件成熟时，agent 应提醒用户：旧路线可能值得重新打开。

完整审计由用户发起或确认；如果出现直接冲突或明显过期的妥协，agent 可以主动提醒。

## 安装

将本仓库作为名为 `semantic-alignment` 的 skill 安装到 Codex 或兼容的 agent 环境中。

skill 根目录结构：

```text
semantic-alignment/
  SKILL.md
  references/
  scripts/
  agents/
```

安装后，在需要长期保持意图一致的项目、设计、实现或写作任务中，让 agent 使用 `semantic-alignment`。

## 记录放在哪里

默认记录目录：

```text
.semantic-alignment/<task-slug>/
```

主要文件包括：

- `user-semantics.md`：给用户查看的当前语义基线
- `user-semantic-ledger.md`：已接受的语义变化和原因
- `recheck-triggers.md`：需要重新检查旧决定的条件
- `realization-semantics.md`：agent 打算实现到产物中的语义
- `artifact-checks.md`：真实产物和实现语义之间的检查
- `audits.md`：对齐、漂移、冲突和建议

这些记录属于项目过程元数据，不是产品本身。
