# User Semantics

This file is the user's review entry point and the current user-semantic baseline. It shows the latest active meaning only; detailed add/update/delete history lives in `user-semantic-ledger.md`.

## Goal

Build an agent skill/package named `semantic-alignment` and publish it to GitHub. It may be packaged first as a Codex-compatible skill, but the semantic record model should remain platform-neutral.

The skill should help project design and implementation stay aligned with the user's real goal as ideas, constraints, realization choices, and artifacts change.

The GitHub package should include a bilingual README and include this project's practice records as examples, while keeping examples separate from runtime skill instructions.

The English and Chinese README files should express the same core semantics: open with a concise, compelling story that AI agents may quietly turn work into something else, explain how the skill prevents semantic drift, remind users to proactively initiate audits at key checkpoints, then explain the solution, audit/reopen questions, trigger workflow, installation, record location, examples, and status. README content should avoid redundant problem sections after the opening hook, avoid foregrounding internal agent script mechanics, and make audit semantics clearer than a loose example list. Chinese copy should be natural but restrained enough for an open-source project README.

## Principles

- Record user semantics, not every user message.
- Keep the user-facing semantic file readable and concise because it is the user's project review entry point.
- Keep current state in `user-semantics.md`; keep semantic history in `user-semantic-ledger.md`.
- Record semantic history as `add`, `update`, or `delete`, with category, before/after state, reason, source, current flag, and recheck trigger.
- `user-semantic-ledger.md` is the action-bearing semantic history. It records accepted user-baseline changes and their recheck triggers.
- Do not keep a separate `semantic-deltas.md`; it duplicates the ledger's practical purpose.
- Agent suggestions or limitations that are not accepted by the user and do not change the route, realization semantics, artifact, or audit finding should not be persisted.
- Do not rely only on agent-written Markdown for structure; use scripts to generate record skeletons and validate required files, headers, and controlled vocabulary.
- Do not bind semantic records to Codex or any single agent platform. Use platform-neutral `.semantic-alignment/` by default.
- Keep `SKILL.md` concise. Move detailed semantic model, operational workflow, and audit rules into references that agents load only when needed.

## Global Design Semantics

- Track user goals, global semantics, local semantics, and reasons for semantic changes.
- Separate user semantics from realization semantics.
- Treat realization semantics as intended artifact semantics: what the agent believes the produced code, design, document, or skill should mean after interpreting user semantics and filling gaps.
- Do not create a separate "artifact semantics" layer; the real artifact is the code/design/document/skill itself, and it is checked against realization semantics.
- Classify agent-filled gaps as `grounded`, `added`, `risky`, or `divergent`.

## Process Semantics

- The agent must read relevant semantic records at required moments; without reading them, it cannot claim alignment, audit readiness, or stale-constraint resolution.
- The agent does not need to reread `user-semantics.md` on every message if a fresh current semantic frame is already loaded in context.
- The current semantic frame should be considered fresh only when it was loaded after the last known semantic-record edit and includes current user semantics, known unknowns, and active recheck triggers.
- Recheck triggers should be generated into a compact current projection file so routine monitoring does not require scanning the full semantic ledger.
- `recheck-triggers.md` should be generated by script from `user-semantic-ledger.md`, preserving per-trigger check method/status notes where possible, instead of relying on agent-maintained trigger rows.
- Recheck triggers must be observable conditions that mean the linked ledger entry should be rechecked; they must not be open tasks, recommendations, or actions.
- Agents must actively evaluate recheck triggers against new user statements, implementation discoveries, artifact checks, tool/dependency/permission/asset changes, delivery checks, and audits. When a trigger appears true, the agent must read the linked ledger row and make the old route plus now-observed condition visible to the user before resolving the trigger or continuing.
- "Actions" are operational rules inside the skill: when to capture, clarify, classify, check, audit, reopen, revise, or escalate.
- `SKILL.md` should provide the minimal mandatory workflow and a clear reference-loading map, not all detailed rules inline.
- `SKILL.md` should include a startup checklist so a fresh agent knows the first actions to take.
- Agents must load the specific reference required for the decision or edit they are about to make.
- New standard/deep record directories should be generated with `scripts/init_records.py`.
- Substantial record edits should be checked with `scripts/validate_records.py`.
- Substantial record edits should also be linted with `scripts/lint_records.py` to catch semantic-quality risks such as action-like triggers or missing current-state projection.
- Agents should prefer `scripts/record_event.py` for appending user-semantic ledger, realization semantics, and artifact-check rows, so IDs and table shape are generated consistently.
- Scripts enforce record shape; agent/user judgment still decides whether semantics are correct.
- During implementation, agents should keep `realization-semantics.md` current as non-obvious implementation choices, inferred details, tradeoffs, or obsolete intended semantics appear, instead of waiting until audit to reconstruct them.
- `user-semantics.md` must stay Markdown because it is user-readable; other record files may move to YAML/JSON/JSONL if stronger parsing becomes useful.
- Selected non-user-entry records may be exported to generated structured JSONL mirrors under `structured/`, while `user-semantics.md` remains the user-readable Markdown entry point.
- The ledger's `Source` and `Recheck trigger` columns carry the practical action hook for future route changes.

## Review Semantics

- Audit user goal, global semantics, and local semantics for internal consistency.
- Audit whether realization semantics faithfully follow user semantics.
- Audit whether the real artifact matches realization semantics.
- Full audits must exhaustively cover every current user semantic and every active realization semantic. User semantic coverage should distinguish satisfied, partial, unmet, conflict, and unknown. Realization semantic coverage should distinguish direct user grounding, aligned extra detail, risky addition, conflict, and unknown grounding.
- Before a full audit, the agent must inspect the real project, update or explicitly confirm `realization-semantics.md`, then refresh or explicitly confirm `artifact-checks.md`. Only the updated realization semantics may be used as audit input.
- Full audits are user-initiated or user-accepted.
- The agent should proactively flag direct user-semantic contradictions, divergent realization semantics, material artifact check failures, or clearly resolved stale constraints.
- When a stale-route recheck trigger becomes true, the agent must make the reminder visible in the user-facing response before resolving the trigger or continuing with the old compromised route.
- If the user says in the same message that the trigger is true and also asks to keep the old compromised route, the agent may treat that as confirmation only when it is explicit, and must still state the stale-route reminder once.
- Repeated audits should keep a current summary plus append-only audit events; old events should be resolved, superseded, accepted, or archived rather than silently deleted.

## Constraint Semantics

- Constraint-driven compromises must be recheckable.
- A constraint record should include blocked route, accepted route, evidence, recheck method, reopen trigger, last checked, and status.
- If a blocking condition disappears, the agent should recommend reviewing whether to reopen the earlier route; material route switches require user confirmation.

## Hygiene Semantics

- Semantic records are project process metadata, not product artifacts.
- Store records per project under `.semantic-alignment/<project-slug>/` by default. The directory may live under a project root or workspace root, but workspace-level storage must use distinct project slugs so multiple projects do not share one semantic baseline.
- Agents should derive `<project-slug>` as a stable project identifier, not a temporary task label: prefer an explicit user-provided project name/slug, then nearest repository root name, then manifest/package project name, then project directory name; normalize to lowercase kebab-case and ask when multiple projects remain plausible.
- Keep semantic records out of source directories, design exports, release artifacts, and user-facing documentation unless explicitly chosen.
- Decide separately whether semantic records belong in Git. Private, noisy, speculative, or internal agent records should usually remain uncommitted.

## Local Semantics

- The skill name is `semantic-alignment`.
- This skill project is the first practice sample for the semantic-alignment workflow.
- This skill project's practice records should be uploaded as examples, not treated as part of the runtime skill.
- Standard records use one task directory with separated files by role.
- Detailed instructions are split into `references/semantic-model.md`, `references/operational-workflow.md`, `references/audit-rules.md`, and `references/semantic-record-template.md`.
- Records should not be centralized in one large file.
- `user-semantics.md` is current state only.
- `user-semantic-ledger.md` is the semantic change and recheck ledger.
- `recheck-triggers.md` is the generated compact current trigger projection derived from `user-semantic-ledger.md`; the ledger remains authoritative history.
- `recheck-triggers.md` groups active ledger entries with the same trigger into one row with comma-separated ledger IDs.
- `scripts/lint_records.py` provides non-blocking semantic-quality warnings; `scripts/validate_records.py` remains the structural validator.
- `scripts/export_structured.py` generates optional JSONL mirrors for selected non-user-entry records.
- `artifact-checks.md` is a mechanical pass/partial/fail check log, not a second audit.
- `audits.md` is the synthesis layer for audit conclusions.
- `realization-semantics.md` and `audits.md` should remain current working records; older non-current realization rows and older resolved/accepted/superseded audit events should be archived when they make routine reads noisy.
- Reference-file splitting should happen only when the resulting files have distinct loading moments; do not split files that agents would still routinely load together.
- Fresh-agent reminder testing showed direct semantic conflict handling is adequate, while stale-route handling needed a stronger user-visible reminder rule.
- GitHub publishing is still open: decide whether the repository should contain only this skill or be part of a multi-skill repository.
- The current GitHub packaging direction is a single-skill repository with `examples/semantic-alignment-skill/` as documentation material.
